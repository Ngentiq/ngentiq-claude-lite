# Commit Message Template

Standard format for conventional commit messages generated by the `/commit` skill.

## Format

```
type(scope): subject line in imperative mood

Optional body explaining WHY this change was made.
Wrap at 72 characters. The diff shows what changed --
the body explains the motivation and context.

Optional footers:
BREAKING CHANGE: description of what breaks and migration path
Refs: #123, #456
Co-Authored-By: Name <email>
```

## Type Reference

| Type | When to Use | Example |
|------|-------------|---------|
| `feat` | New feature or capability | `feat(auth): add OAuth2 login flow` |
| `fix` | Bug fix | `fix(api): handle null response from payment gateway` |
| `refactor` | Code restructuring without behavior change | `refactor(db): extract query builder from repository` |
| `test` | Adding or updating tests | `test(auth): add integration tests for token refresh` |
| `docs` | Documentation changes | `docs(readme): add deployment instructions` |
| `chore` | Maintenance, dependencies, tooling | `chore(deps): update eslint to v9` |
| `perf` | Performance improvement | `perf(search): add index for full-text queries` |
| `style` | Formatting, whitespace, semicolons | `style: apply prettier formatting` |
| `ci` | CI/CD configuration | `ci: add Node 22 to test matrix` |
| `build` | Build system or external dependencies | `build: migrate from webpack to vite` |

## Rules

- **Subject line**: imperative mood, no period at end, max 72 characters total (including type and scope)
- **Scope**: lowercase, the primary area affected (e.g., `auth`, `api`, `cli`, `config`, `db`)
- **Body**: separated from subject by a blank line, wrapped at 72 characters, explains motivation not mechanics
- **Breaking changes**: use `BREAKING CHANGE:` footer (not `!` suffix) for clarity
- **Multiple concerns**: if a commit touches multiple areas, use the most significant one as the scope

## Examples

### Simple feature
```
feat(search): add fuzzy matching for product names
```

### Bug fix with context
```
fix(payments): prevent double-charge on retry

The payment gateway returns 200 on duplicate idempotency keys but
does not actually charge again. Our retry logic was treating this as
a new charge and updating the ledger twice.
```

### Breaking change
```
feat(api): require API key header on all endpoints

Previously, read-only endpoints were unauthenticated. All endpoints
now require the X-API-Key header.

BREAKING CHANGE: all API requests must include X-API-Key header.
Clients using read-only endpoints without authentication will
receive 401 responses.
```

### With co-author
```
refactor(db): consolidate migration runner into single entry point

Co-Authored-By: Claude <noreply@anthropic.com>
```
