# Plan Output Template

This is the output format for feature plan documents generated by `/plan`. The plan is written to `plans/P-{NNN}-{slug}/plan.md`.

---

```markdown
# Feature Plan: {Feature Title}

> Generated by /plan on {date}

## Overview

{2-4 sentence summary of what the feature does and why it matters. Include the user-facing value and the technical approach at a high level.}

## Requirements

| # | Requirement | Type | Priority |
|---|-------------|------|----------|
| R1 | {requirement} | functional | must-have |
| R2 | {requirement} | functional | must-have |
| R3 | {requirement} | non-functional | should-have |

Type: functional, non-functional, constraint
Priority: must-have, should-have, nice-to-have

## Out of Scope

{Explicitly list what this feature does NOT include. This prevents scope creep during implementation and sets clear boundaries for reviewers.}

- {Item explicitly excluded and why}

## Business Acceptance Criteria

These criteria define when the feature is "done" from a product/stakeholder perspective. Written in user-facing language.

| # | Criterion | Requirement | Verification Method |
|---|-----------|-------------|---------------------|
| BAC-1 | {User-facing outcome in plain language} | R1 | {How a stakeholder verifies this} |
| BAC-2 | {Business rule enforcement} | R2 | {How a stakeholder verifies this} |
| BAC-3 | {User experience expectation} | R1, R3 | {How a stakeholder verifies this} |

Every requirement MUST have at least one BAC item. BAC items use stakeholder language, not engineering language.

## QA Acceptance Criteria

These criteria define the quality gates the feature must pass. Written in engineering language.

| # | Criterion | Type | Requirement | Automated |
|---|-----------|------|-------------|-----------|
| QAC-1 | {Functional test scenario} | unit | R1 | yes |
| QAC-2 | {Edge case / error handling} | integration | R1 | yes |
| QAC-3 | {Non-functional requirement} | performance | R3 | yes/no |
| QAC-4 | {Security scenario} | security | R2 | yes |

Type: unit, integration, e2e, performance, security
Every requirement MUST have at least one QAC item. QAC items are specific enough to write a test from.

## Impact Analysis

### Affected Files

| File | Change Type | Description |
|------|-------------|-------------|
| {file path} | modify | {what changes and why} |
| {file path} | create | {what this new file does} |
| {file path} | delete | {why this file is removed} |

Change Type: create, modify, delete

### Architecture / Design

{How the feature fits within the existing architecture. Key design decisions and their rationale. Patterns from the existing codebase that will be followed. Any new patterns being introduced and why.}

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| {risk description} | low/medium/high | low/medium/high | {mitigation strategy} |

## Implementation Plan

### Phase 1: {Phase Name}

| Step | Description | Files | Dependencies |
|------|-------------|-------|--------------|
| 1 | {what to do} | {files to touch} | none |
| 2 | {what to do} | {files to touch} | step 1 |

{Break into phases only if the feature warrants it. Small features use a single unnumbered phase.}

## Test Plan

### Unit Tests

| Test | Target | QAC | Scenario |
|------|--------|-----|----------|
| {test description} | {file/module} | QAC-1 | {what is being verified} |

### Integration Tests

| Test | Components | QAC | Scenario |
|------|-----------|-----|----------|
| {test description} | {components involved} | QAC-2 | {what is being verified} |

{Include only the test categories relevant to this feature: unit, integration, e2e, performance.}

## Open Questions

- {Ambiguity or decision that needs resolution before implementation}
- {Missing information that affects scope or approach}

{Omit this section if there are no open questions.}
```
